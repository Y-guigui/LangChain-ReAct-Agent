## 1. RAG 检索增强生成

### 知识点

**RAG (Retrieval-Augmented Generation)** 是一种结合"检索"和"生成"的技术架构，核心思想是：

> 大模型回答问题时，先到知识库中检索相关文档，再将检索结果作为上下文提供给 LLM 生成答案。

```
用户问题 → [向量检索] → 知识库相关文档 → [LLM生成] → 回答
```

**为什么需要 RAG？**
- LLM 知识有截止日期，无法回答新知识
- LLM 容易产生幻觉（编造不存在的"事实"）
- 企业内部知识 LLM 从未见过

**核心流程：**

```
1. 文档加载    → 读取 PDF/TXT/Markdown 文件
2. 文本分割    → 将长文档切成小块（chunk）
3. 向量嵌入    → 每块文本转为向量（embedding）
4. 向量存储    → 存入向量数据库（Chroma）
5. 相似检索    → 用户问题转为向量，找最相似的 K 个文档块
6. LLM 生成    → 将检索到的文档作为上下文，让 LLM 生成回答
```

### 代码实现

**步骤1：文档加载** (`utils/file_handler.py`)

**步骤2-4：文本分割 + 向量化** (`rag/vector_store.py`)

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

class VectorStoreService:
    def __init__(self):
        # 创建文本分割器：每200字符一块，前后重叠20字符
        self.spliter = RecursiveCharacterTextSplitter(
            chunk_size=200,      # 每块200个字符
            chunk_overlap=20,    # 相邻两块重叠20字符（避免上下文断裂）
            separators=["\n\n","。",".","?","？","!"," ",""],  # 优先在句子处断开
        )
        # 向量数据库：Chroma
        self.vector_store = Chroma(
            collection_name="agent",
            embedding_function=embed_model,    # text-embedding-v4
            persist_directory="chroma_db"       # 持久化到磁盘
        )

    def load_documents(self):
        # 1. 扫描知识库目录
        files = listdir_with_allowed_type("data/knowledge", ("txt", "pdf", "md"))
        # 2. 逐个文件加载
        for path in files:
            documents = get_file_documents(path)     # 加载文档
            split_docs = self.spliter.split_documents(documents)  # 切片
            self.vector_store.add_documents(split_docs)           # 存入向量库
```

**为什么要有 chunk_overlap？**

```
原文: "Python的列表是可变的，元组不可变。元组可以作为字典的key。"
分块:  [--------块1:200字符--------]
                  [--------块2:200字符--------]  ← 重叠20字符
```
重叠避免关键信息恰好被切在块边界导致丢失。

**步骤5：相似检索**

```python
def search(self, query: str, top_k: int = 3) -> list[Document]:
    """将用户问题转为向量，找最相似的top_k个文档块"""
    retriever = self.vector_store.as_retriever(
        search_kwargs={"k": top_k}
    )
    return retriever.invoke(query)
```

**步骤6：RAG 生成** (`rag/rag_service.py`)


## 2. ReAct Agent 模式

### 知识点

**ReAct = Reasoning + Acting**，即"思考 → 行动 → 观察 → 再思考"的循环模式。

传统 LLM 对话：用户问 → LLM 答（一次完成）
ReAct Agent：用户问 → LLM 思考 → 调用工具 → 观察结果 → 再思考 → 最终回答

```
┌─────────────────────────────────────────────────┐
│ Step 1: 用户: "帮我查一下昨天聊过的Python知识点"                 │
│ Step 2: Thought: "用户提到了'昨天聊过'，需要先搜索记忆"            │
│ Step 3: Action: search_memory("Python知识点")                     │
│ Step 4: Observation: "1. [fact] 用户正在学习Python..."            │
│ Step 5: Thought: "记忆已获取，可以结合知识库给出回答了"              │
│ Step 6: Answer: "根据你之前的记录，你正在学习..."                   │
└─────────────────────────────────────────────────┘
```

**核心优势**: Agent 可以自主决定调用哪些工具、以什么顺序调用、需要调用几次。

### 代码实现 (`agent/react_agent.py`)

```python
from langchain.agents import create_agent

class ReactAgent:
    def __init__(self):
        # create_agent 是 LangChain 提供的 ReAct Agent 创建函数
        self.agent = create_agent(
            model=chat_model,           # LLM 模型（qwen3.7-max）
            system_prompt=system_prompt, # 系统提示词
            tools=[                      # 4个工具
                rag_summarize,           # RAG知识库检索
                get_current_time,        # 获取当前时间
                search_memory,           # 搜索长期记忆
                store_memory,            # 存储长期记忆
            ],
            middleware=[                # 2个中间件
                log_before_model,       # 记录模型调用前状态
                report_prompt_switch,   # 动态切换提示词
            ],
        )

    def execute_stream(self, query: str, messages=None, session_id=None):
        # 构建输入消息
        input_dict = {"messages": [{"role":"user","content": query}]}

        # 流式执行 Agent
        for chunk in self.agent.stream(
            input_dict,
            stream_mode="values",
            context={"report": False}
        ):
            latest = chunk["messages"][-1]
            if latest.content:
                yield latest.content.strip() + "\n"
```

## 3. 多轮对话上下文管理（短期记忆）

### 知识点

**多轮对话**的核心挑战：LLM 没有记忆，每次请求都是独立的。

```
第1轮: 用户: "Python 中 list 和 tuple 的区别是什么？"
       助手: "list 可变，tuple 不可变..."
第2轮: 用户: "那它在 Django 开发中怎么选择？"  ← "它"指什么？
```

如果不传上下文，LLM 不知道"它"指的是list/tuple。所以需要：

1. **会话管理** — 每轮对话关联同一个 session_id
2. **消息历史** — 将之前几轮的对话拼到 prompt 中
3. **状态跟踪** — 识别意图、提取实体、管理槽位

### 代码实现

**会话模型** (`context/schemas.py`)

```python
class Session(BaseModel):
    session_id: str                # 会话唯一标识
    messages: List[Message]        # 消息历史
    state: DialogueState           # 对话状态
    created_at: datetime           # 创建时间
    updated_at: datetime           # 最后活跃时间

    def get_recent_messages(self, n=10) -> List[Message]:
        """获取最近n条消息，作为上下文传给LLM"""
        return self.messages[-n:]
```

**会话管理器** (`context/session_manager.py`)

```python
class SessionManager:
    def create_session(self) -> str:

        self._sessions[session.session_id] = session
        return session.session_id

    def add_message(self, id: str, role: str, content: str):
        """向会话中添加一条消息（用户提问/助手回复）"""
        session = self.get_session(id)
        session.add_message(role, content)

    def get_recent_context(self, id: str, n=10) -> list[dict]:
        """获取会话的最近上下文，供Agent使用"""

        return session.to_langchain_messages()[-n:]
```

## 4. 长期记忆系统

### 知识点

**短期记忆 vs 长期记忆** 是 Agent 系统的两个核心记忆维度：

| 维度 | 短期记忆（Short-term） | 长期记忆（Long-term） |
|------|----------------------|----------------------|
| **存储内容** | 当前会话的对话历史 | 用户的偏好、事实、目标、反馈 |
| **生命周期** | 会话关闭即销毁 | 跨会话持久化 |
| **存储方式** | Python 内存（dict） | SQLite 元数据 + Chroma 向量 |
| **检索方式** | 按时间顺序取最近N条 | 语义相似度搜索 |
| **典型场景** | "刚才说到哪了？" | "帮我回忆一下我之前学 Python 的进度" |

**为什么需要长期记忆？**

```
会话 #1: 用户: "我叫阿彤，正在学 Python。"
         Agent: "好的，阿彤！"

会话 #2（新会话，无上下文）:
        用户: "帮我推荐学习资源。"
        期望回答: "阿彤，根据你学 Python 的进度..."
        没有长期记忆: "你好！请问怎么称呼？"
```

没有长期记忆，Agent 每次都是"第一次见面"。有了长期记忆，Agent 能记住用户是谁、做过什么、偏好什么。

**双重存储架构（SQLite + Chroma）：**

```
添加记忆:
  Memory {content, category, importance}
        │
    ┌───┴───┐
    │       │
  SQLite   Chroma
  (元数据)  (向量)
    │       │
    └───┬───┘
  双写完成

检索记忆:
  用户 query → 向量化 → Chroma 语义搜索
                        ↓
                   候选结果 → SQLite 元数据过滤
                        ↓
                   最终结果（按相似度排序）
```

**五类记忆分类：**

| 分类 | 英文 | 示例 |
|------|------|------|
| 偏好 | `preference` | "喜欢简洁的回答风格" |
| 事实 | `fact` | "用户名叫阿彤，正在学 Python" |
| 目标 | `goal` | "计划三个月内完成 Python 面试准备" |
| 反馈 | `feedback` | "上次关于 GIL 的解释很清楚" |
| 性格 | `personality` | "用户偏好先看结论再看解释" |

**记忆提取（LLM 驱动）：**

不依赖人工手动录入记忆。Agent 在每轮对话结束后，自动用 LLM 分析对话内容，提取有价值的信息存入记忆。

```
对话轮次 → MemoryExtractor (LLM) → 提取记忆 → MemoryStore (双写)
```

### 代码实现

**记忆数据模型** (`memory/schemas.py`)

```python
class Memory(BaseModel):
    memory_id: Optional[str] = Field(default=None, description="记忆唯一ID")
    content: str = Field(..., description="记忆内容")
    category: str = Field(default="fact", description="分类")   # 5种分类
    importance: float = Field(default=0.5, ge=0.0, le=1.0, description="重要性评分")
    source: str = Field(default="", description="来源")
    created_at: datetime = Field(default_factory=datetime.now)

class MemoryQuery(BaseModel):
    query: str = Field(..., description="查询内容")
    top_k: int = Field(default=5, ge=1, le=20, description="返回数量")
    category: Optional[str] = Field(default=None, description="按分类过滤")
    min_importance: float = Field(default=0.0, ge=0.0, le=1.0, description="最小重要性")
```

**记忆存储 — SQLite + Chroma 双写** (`memory/store.py`)

```python
class MemoryStore:
    """长期记忆存储（SQLite + Chroma 双写）"""

    def __init__(self, db_path=None, persist_dir=None):
        # SQLite 存储元数据（内容、分类、重要性、来源、时间）
        self.db_path = db_path or get_abs_path("chroma_db/memory.sqlite")

        # Chroma 存储向量（语义检索）
        self.vector_store = Chroma(
            collection_name="long_term_memory",
            embedding_function=embed_model,
            persist_directory=persist_dir or get_abs_path("chroma_db"),
        )
        self._init_db()  # 建表 + 索引

    def add(self, memory: Memory) -> str:
        """添加记忆：SQLite 写元数据 + Chroma 写向量"""
        memory_id = str(uuid.uuid4())[:8]

        # 1. SQLite — 写入可查询的元数据（category 和 importance 有索引）
        conn = sqlite3.connect(self.db_path)
        conn.execute(
            "INSERT INTO memories (memory_id, content, category, importance, source, created_at) "
            "VALUES (?, ?, ?, ?, ?, ?)",
            (memory_id, memory.content, memory.category,
             memory.importance, memory.source, memory.created_at.isoformat()),
        )
        conn.commit(); conn.close()

        # 2. Chroma — 将内容向量化存储
        doc = Document(page_content=memory.content, metadata={
            "memory_id": memory_id, "category": memory.category,
            "importance": memory.importance, "source": memory.source,
        })
        self.vector_store.add_documents([doc])
        return memory_id

    def search(self, query: MemoryQuery) -> List[Memory]:
        """搜索记忆：向量相似度 + 元数据过滤"""
        # 1. Chroma 语义检索（按内容语义相似度排序）
        retriever = self.vector_store.as_retriever(search_kwargs={"k": query.top_k})
        docs = retriever.invoke(query.query)

        # 2. 过滤（按分类和重要性二次筛选）
        results = []
        for doc in docs:
            meta = doc.metadata
            if query.category and meta.get("category") != query.category:
                continue  # 分类不匹配，跳过
            if meta.get("importance", 0) < query.min_importance:
                continue  # 重要性不足，跳过
            results.append(Memory(
                memory_id=meta.get("memory_id"), content=doc.page_content,
                category=meta.get("category", "fact"),
                importance=meta.get("importance", 0.5),
                source=meta.get("source", ""),
            ))
        return results
```

**为什么用双写而不是纯 Chroma？**

Chroma 天然支持元数据存储，为什么还要用 SQLite？
- SQLite 支持高效的分类查询和按时间排序（`SELECT ... WHERE category = ? ORDER BY created_at DESC`）
- Chroma 适合语义检索但不够擅长关系型查询
- SQLite 统计信息更直观：`SELECT category, COUNT(*) FROM memories GROUP BY category`
- 两者互补，各取所长

**记忆提取器 — LLM 驱动** (`memory/extractor.py`)

```python
class MemoryExtractor:
    """从对话中自动提取值得长期记忆的信息"""

    def __init__(self):
        self.prompt = PromptTemplate.from_template(EXTRACTION_PROMPT)
        self.chain = self.prompt | chat_model

    def extract(self, user_message: str, assistant_message: str) -> List[dict]:
        """
        从一轮对话中提取记忆

        :return: [{"content": "用户偏好简洁回答", "category": "preference", "importance": 0.8}]
        """
        dialogue = f"用户：{user_message}\n\n助手：{assistant_message}"
        result = self.chain.invoke({"dialogue": dialogue})
        text = result.content  # LLM 返回的 JSON 字符串

        # 解析 LLM 输出的 JSON
        start, end = text.find("["), text.rfind("]")
        if start >= 0 and end > start:
            memories = json.loads(text[start:end+1])
            return memories  # → [{"content": "...", "category": "...", "importance": 0.8}]
        return []
```

**提取的 Prompt 模板（核心精华）：**

```text
请从以下对话中提取关键信息，判断是否有值得长期记忆的内容。

需要提取的信息类型：
1. 用户偏好（preference）：用户明确表达的喜好、习惯、偏好设置
2. 事实信息（fact）：用户提到的个人事实、背景、经历
3. 目标/计划（goal）：用户提到的学习目标、工作计划、待办事项
4. 用户反馈（feedback）：用户对系统、产品的评价或建议
5. 性格/习惯（personality）：用户展现出的性格特点或工作习惯

对话内容：{dialogue}

请以 JSON 格式输出：[{"content": "记忆内容", "category": "...", "importance": 0.8}]
如果没有值得记忆的信息，输出空数组 []。
```

```python
    return "\n".join(lines)

@tool(description="将关键信息存入长期记忆。入参content为记忆内容，category为分类，importance为重要性（0-1）。")
def store_memory(content: str, category: str = "fact", importance: float = 0.5) -> str:
    """存储记忆：Agent 自主决定何时调用"""
    memory = Memory(content=content, category=category, importance=importance, source="对话")
    memory_id = memory_store.add(memory)
    return f"记忆已保存 [{memory_id}]: {content[:50]}..."
```

**完整的记忆生命周期：**

```
1. 用户发言
      ↓
2. ReAct Agent 分析意图
   → 如果是"回忆类"问题，调用 search_memory 工具
   → 如果是新事实/偏好，调用 store_memory 工具
      ↓
3. SSE 流式返回回答
      ↓
4. 对话结束后，MemoryExtractor 用 LLM 自动提取记忆
      ↓
5. SQLite + Chroma 双写持久化
```

## 5. 评估体系

### 知识点

RAG 系统的质量需要量化评估——你改了检索策略后，怎么知道变好了还是变坏了？

**六大核心指标：**

| 指标 | 计算方式 | 含义 |
|------|---------|------|
| **Recall@K** | 前K个结果中相关文档数 / 总相关文档数 | 找全了吗？ |
| **Precision@K** | 前K个结果中相关文档数 / K | 找对了吗？ |
| **MRR** | 第1个相关文档排名的倒数 | 排得靠前吗？ |
| **NDCG@K** | 对排序质量加权打分 | 排得准吗？ |
| **F1 Score** | 2*P*R/(P+R) | 综合平衡 |
| **Hit Rate** | 是否至少命中一个 | 有没有？ |

### 代码实现

```python
def recall_at_k(retrieved_ids, relevant_ids, k=None):
    """Recall@K = 命中数 / 应有数"""
    if k: retrieved_ids = retrieved_ids[:k]
    hits = sum(1 for doc_id in retrieved_ids if doc_id in relevant_ids)
    return hits / len(relevant_ids)

def mrr(retrieved_ids, relevant_ids):
    """MRR = 1 / 第一个相关文档的位置"""
    for i, doc_id in enumerate(retrieved_ids, start=1):
        if doc_id in relevant_ids:
            return 1.0 / i
    return 0.0

def ndcg_at_k(retrieved_ids, relevant_ids, k=None):
    """NDCG = DCG / IDCG（考虑排序位置加权）"""
    scores = [1 if doc_id in relevant_ids else 0 for doc_id in retrieved_ids]
    dcg = sum((2**s-1)/log2(i+2) for i,s in enumerate(scores))
    ideal = sorted(scores, reverse=True)
    idcg = sum((2**s-1)/log2(i+2) for i,s in enumerate(ideal))
    return dcg/idcg if idcg>0 else 0.0
```

> 📂 对应项目中文件：`evaluation/metrics.py`、`evaluation/evaluator.py`、`evaluation/benchmark.py`

## 6. FastAPI 工程架构

### 知识点

本项目采用**分层架构**设计，从 Streamlit 单体应用改为 FastAPI 模块化架构：

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  前端 (static)│ ←→ │  API层 (api/) │ ←→ │ 业务层       │

```

**为什么分层？**
- 前端改UI不改后端代码
- API 改接口不改业务逻辑

```python
    for chunk in agent.execute_stream(query, messages=messages):
```
