> 本文档针对简历中 4 条核心技术点，逐条拆解设计思路、名词解释和可能的面试追问。建议面试前通读 2-3 遍，用你自己的话复述关键设计决策。

---

## 技术点 1：RAG 检索引擎（Chroma + text-embedding-v4）

> 简历原文：*基于 Chroma + text-embedding-v4 构建 RAG 检索引擎，实现 txt/pdf/md 文档导入、文本分片、MD5 去重与语义相似度检索，解决了大模型回答容易脱离资料胡编的问题，回答具备可追溯的引用依据。*

---

### 1.1 设计思路

**问题**：大模型（LLM）有两个致命缺陷——知识有截止日期，以及"幻觉"（凭空编造不存在的事实）。直接问 LLM 一个需要特定资料才能回答的问题，它可能给出看似合理实则错误的答案。

**方案**：RAG（Retrieval-Augmented Generation，检索增强生成）的核心思想是"先检索，再生成"——在 LLM 回答之前，先到知识库中搜索相关文档，把检索到的内容作为"参考资料"塞进 Prompt，让 LLM 基于这些真实资料来回答。

```
传统模式：用户提问 → LLM（凭记忆瞎编） → 可能胡说的回答

RAG 模式：用户提问 → 向量检索 → 找到相关文档块
                    ↓
              文档块 + 问题 → LLM（基于资料） → 有据可查的回答
```

**完整流程（6 步）：**

```
1. 文档加载    → TextLoader / PyPDFLoader 读取 txt/pdf/md
2. 文本分割    → RecursiveCharacterTextSplitter（chunk_size=200, overlap=20）
3. 向量嵌入    → text-embedding-v4 将文本块转为 1024 维向量
4. 向量存储    → 写入 Chroma 向量数据库（persist_directory 持久化到磁盘）
5. 相似检索    → 用户问题向量化 → 余弦相似度 → Top-K 最相关文档块
6. LLM 生成    → 检索结果拼接 Prompt → qwen3.7-max 生成回答
```

---

### 1.2 名词解释

**Embedding（向量嵌入）**
把一段文本（词、句子、段落）转换成一串浮点数（向量）。语义相似的文本，它们的向量在空间中的距离就更近。比如 "Python 是什么" 和 "Python 语言简介" 的向量距离很近，而和 "今天天气不错" 的向量距离很远。向量之间的"距离"通常用余弦相似度来衡量。

**text-embedding-v4**
阿里云百炼平台提供的文本嵌入模型，输入文本 → 输出 1024 维向量。选它是因为和项目用的通义千问 LLM 是同一个生态，API 调用一致性好，而且中文语义理解能力不错。

**Chroma**
开源的向量数据库。类比：MySQL 存的是"行和列"，Chroma 存的是"向量和元数据"。支持本地持久化（不需要单独部署服务器），内置了相似度检索（`collection.query()`），还支持元数据过滤。选它是因为轻量、Python SDK 友好，适合个人项目而非需要 Milvus/Pinecone 等重量级方案。

**RecursiveCharacterTextSplitter（递归字符分割器）**
LangChain 提供的文本分割工具。不是简单按长度切，而是按优先级逐步尝试分割：先尝试按段落（`\n\n`）切，切不动就按句号切，再按空格切。这样尽量保证每个 chunk 是一个完整的语义单元，不会把一句话拦腰截断。

**chunk_size=200, chunk_overlap=20**
每个文本块 200 字符，相邻两块重叠 20 字符。重叠的目的是防止关键信息恰好落在块边界上。比如原文是 "Python 的 GIL 保证了引用计数的线程安全"，如果切割点恰好在 "GIL" 和 "保证了" 之间，重叠 20 个字符就能让这块信息在两个 chunk 中都出现，不会丢。

**MD5 去重**
上传文件时计算文件内容的 MD5 哈希值，存入 md5.txt 记录。下次再上传相同内容的文件，比对 MD5 发现相同就跳过，避免向量库中出现重复数据。这比直接比较文件名或文件大小更可靠。

**语义相似度检索**
用户提问"GIL 是什么影响"，被转成向量后，Chroma 在向量空间中计算这个向量与所有文档向量的余弦相似度，返回 Top-K 个最接近的文档块。和关键词匹配不同——关键词匹配搜"GIL"只返回含"GIL"的文档，向量检索搜"Python 全局解释器锁"也能找到。

### 1.3 面试追问及回答要点

**Q：RAG 和微调（Fine-tuning）有什么区别？为什么选 RAG？**

> RAG 是"外挂知识库"，LLM 本身不改变，只在推理时动态检索外部知识注入 Prompt。微调是"把知识烧进模型"，通过训练改变模型权重。
>
> 选 RAG 的原因：① 知识可以随时更新（新增/删除文档即可，不需要重训）；② 可追溯——回答的每句话都能定位到具体的参考资料；③ 成本低——微调需要 GPU 算力和标注数据，RAG 只需要向量库。
>
> 实际中两者不互斥，可以做 RAG + 微调的混合方案。但个人项目场景下 RAG 性价比更高。

**Q：为什么选 Chroma 而不是 Milvus / Pinecone / FAISS？**

> 选型标准是"个人项目需要什么"——不需要分布式、不需要高并发、不想额外部署一个数据库服务。Chroma 是 Python 原生库，`pip install` 就能用，本地文件持久化，API 简洁（`add_documents` / `as_retriever`），而且和 LangChain 深度集成。
>
> FAISS 是 Meta 的向量检索库，纯粹做检索不提供存储管理，需要自己管理文档-向量映射。Milvus 功能强但需要 Docker 部署，对个人项目太重。Pinecone 是云服务要付费。

**Q：chunk_size 为什么选 200？怎么确定这个值的？**

> 没有"唯一正确答案"，200 是一个实验后的经验值。chunk 太小（比如 50）→ 语义碎片化，一段完整论述被切散，检索召回后 LLM 看到的上下文不连贯。chunk 太大（比如 1000）→ 语义稀释，检索精度下降，而且 Token 消耗增大。
>
> 200 是参考 LangChain 官方建议 + 结合知识库内容密度（Python 面试题，每 200 字通常是一条独立的小知识点）决定的。项目中评估体系的存在就是为了验证这类参数——改了 chunk_size 跑一遍评估，看 Recall 和 MRR 变化就知道效果。

**Q：如果检索结果都不相关怎么办？有没有兜底策略？**

> 项目中有两层兜底：① 在 RAG 链中检查检索结果的相似度阈值，如果最高相似度也低于某个值，判断为"知识库无相关内容"，触发回退逻辑；② 回退到纯 LLM 回答模式，但在回答中明确告知用户"知识库中未找到相关信息，以下回答基于通用知识"。
>
> 同时也会记录这类"未命中"案例，作为知识库补充的参考——哪些问题用户问了但知识库没有，后续针对性补充文档。

## 技术点 2：长期记忆系统（SQLite + Chroma 双写 + MemoryExtractor）

> 简历原文：*设计 SQLite（元数据）+ Chroma（向量）双重存储的长期记忆系统，配合 LLM 驱动的 MemoryExtractor 自动从对话中提取用户偏好/事实/目标等信息，解决了每次对话都是"初次见面"的上下文断裂问题，实现跨会话的用户状态记忆。*

---

### 2.1 设计思路

**问题**：LLM 每次请求都是无状态的。用户第 1 轮说"我叫阿彤，正在学 Python"，第 2 轮说"帮我推荐学习资源"，如果不传记忆，LLM 不知道用户是谁、学过什么。

**方案**：设计一个跨会话持久化记忆系统。每次对话结束后，用 LLM 自动分析对话内容，提取"值得记住的事"，存入数据库。下次对话开始时，根据用户提问语义搜索相关记忆，注入 Prompt。

**核心设计：为什么是 SQLite + Chroma 双写？**

```
单用 Chroma → 擅长语义检索，但分类查询差：
  "列出所有 preference 类型的记忆" → 做不到高效的 WHERE 过滤

单用 SQLite → 擅长结构化查询，但语义检索差：
  "搜和 Python 学习相关的记忆" → LIKE '%Python%' 太粗糙

双写方案 → 各取所长：
  写入：SQLite（结构化的行） + Chroma（语义化的向量）同步写入
  检索：先用 Chroma 做语义 Top-K 召回 → 再用 SQLite 元数据过滤 → 最终结果
```

**五类记忆分类的设计考量：**

| 分类 | 触发场景 | 为什么需要 |
|------|---------|-----------|
| `preference` | 用户说"我喜欢简洁的回答" | 个性化回答风格 |
| `fact` | 用户说"我叫阿彤，刚入职" | 记住用户身份和背景 |
| `goal` | 用户说"想三个月内通过面试" | 主动提醒和跟进 |
| `feedback` | 用户说"上次的解释很清楚" | 记录交互质量，优化后续 |
| `personality` | 从多轮对话推断用户习惯 | 调整回答方式 |

**MemoryExtractor 的设计动机**：

不能靠人工手动录入记忆，用户体验太差。设计了一个 Prompt 模板让 LLM 自动分析每轮对话，判断：① 这段话有没有值得长期记忆的信息？② 如果有，属于什么分类？③ 重要程度有多高？LLM 输出 JSON 格式的记忆列表，程序解析后调用 `MemoryStore.add()` 双写存储。

---

### 2.2 名词解释

**Memory（记忆实体）**
一条记忆由 6 个字段组成：
- `memory_id`：唯一标识（UUID 前 8 位）
- `content`：记忆内容（全文，如"用户名叫阿彤，目前正在学习 Python，目标是三个月通过面试"）
- `category`：分类（preference / fact / goal / feedback / personality）
- `importance`：重要性评分（0.0 ~ 1.0），用于检索时过滤低价值记忆
- `source`：来源（对话ID或会话ID）
- `created_at`：创建时间戳

**MemoryQuery（记忆查询）**
搜索记忆时不是简单传一个关键词，而是一个结构化查询对象：
- `query`：语义搜索文本（会先向量化再检索）
- `top_k`：返回数量上限
- `category`：可选，只搜索指定分类
- `min_importance`：可选，过滤掉重要性太低的记忆

**双写一致性**
写入时，SQLite 和 Chroma 各写一次。如果其中一个失败了怎么办？当前实现是 SQLite 先写（同步），然后 Chroma 写。如果 Chroma 失败，日志会记录警告，但 SQLite 中已经有数据了。这是一个权衡——对个人项目来说，偶尔的不一致可以接受，不需要分布式事务。

**MemoryExtractor（记忆提取器）**
核心是一段精心设计的 Prompt：
```
请从以下对话中提取关键信息，判断是否有值得长期记忆的内容。
需要提取的信息类型：
1. 用户偏好（preference）
2. 事实信息（fact）
3. 目标/计划（goal）
...
请以 JSON 格式输出：[{"content": "...", "category": "...", "importance": 0.8}]
如果没有值得记忆的信息，输出空数组 []。
```

Prompt 设计的要点：
- 明确 5 种分类的含义（让 LLM 知道什么算 preference、什么算 fact）
- 要求 JSON 格式输出（便于程序解析）
- 允许空数组（不是每轮对话都有记忆价值）
- 要求每个记忆带 importance（让 MemoryQuery 的 min_importance 过滤有效）

### 2.3 面试追问及回答要点

**Q：为什么不直接用 Chroma 自带的 metadata 存储，非要加 SQLite？**

> Chroma 确实支持 metadata 存储和过滤，但它是向量数据库的"副业"，不擅长复杂的关系型查询。具体来说：① SQLite 支持 `SELECT COUNT(*) FROM memories GROUP BY category` 做分类统计，Chroma 做不了 GROUP BY；② SQLite 支持 `ORDER BY created_at DESC` 按时间排序，Chroma 的排序逻辑围绕向量距离而非时间；③ SQLite 支持事务和批量 UPDATE/DELETE，Chroma 的更新操作相对受限；④ 万一有一天想换向量数据库（比如迁移到 Milvus），记忆的元数据在 SQLite 中不受影响。
>
> 本质上这是一个"存储分离"的设计思路：向量检索交给 Chroma，结构化查询交给 SQLite。

**Q：MemoryExtractor 的提取质量怎么保证？如果 LLM 提取错了怎么办？**

> 这是一个需要诚实回答的问题——当前没有自动质检。LLM 确实可能提取出错（把闲聊当事实、给不重要的信息打高分）。在 Prompt 层面做了尽可能的约束（明确 5 分类定义 + 要求 JSON 格式 + 允许空数组），但本质上依赖 LLM 的理解能力。
>
> 实际的改进方向：① 让用户在下次对话中隐式纠正（用户说"不对，我不是...", `store_memory` 覆盖旧记忆）；② 加入人工审核环节（标记待确认的记忆）；③ 用更强的模型做提取（当前用的是同一套 qwen3.7-max，可以尝试用更好的推理模型专门做提取）。这些都是可以在面试中聊的"如果让我继续做"的分层改进点。

**Q：如果用户说了很多，产生大量记忆，语义检索会不会越来越慢？怎么控制记忆膨胀？**

> 记忆膨胀确实是一个潜在问题。当前项目有几层控制：① `importance` 评分过滤——检索时可以设置 `min_importance=0.5`，过滤掉琐碎记忆；② `top_k` 限制返回数量；③ Chroma 的向量检索本身是 O(log n) 的近似最近邻搜索，数据量增长到十万级别也不会明显变慢。
>
> 如果要进一步优化：可以定期运行"记忆去重"任务（合并重复或相似的记忆）；设置记忆过期时间（长时间未访问的低重要性记忆自动清理）；使用重要性衰减机制（久未使用的记忆 importance 逐步降低）。

**Q：长期记忆和短期记忆的关系是什么？在你的项目中怎么配合的？**

> 短期记忆是"这一轮对话"的上下文（SessionManager），存的是最近的对话历史，会话关闭就销毁。长期记忆是"这个用户"的历史（MemoryStore），跨会话持久化。
>
> 配合方式：① Agent 收到提问后，先用 `search_memory` 工具从长期记忆中检索相关历史信息；② 同时 SessionManager 提供当前会话的最近消息作为上下文；③ 两者拼接后注入 Agent 的推理上下文——长期记忆提供"你是谁、你之前做过什么"，短期记忆提供"你刚才在聊什么"。
>
> 这是一个典型的双层记忆架构，灵感来自人类的记忆模型——工作记忆（短期）+ 长期记忆。

## 技术点 3：短期记忆系统（SessionManager + DialogueTracker）

> 简历原文：*实现基于 SessionManager + DialogueTracker 的短期记忆系统，支持会话超时自动过期、意图识别（知识检索/记忆查询/一般问答）与槽位填充，配合长期记忆形成完整的双层记忆架构。*

---

### 3.1 设计思路

**问题**：HTTP 是无状态协议。用户发多条消息，每次请求都是一个独立的 HTTP POST。如果不做会话管理，LLM 收到的永远是"孤立的最后一句话"，不知道前面聊了什么。

**方案**：在内存中维护一个"会话池"（`{session_id: Session对象}`），每个会话对象包含：
- 完整的消息历史（用户和助手的所有对话）
- 对话状态（当前意图、已提取的实体、已填充的槽位）
- 生命周期管理（创建时间、过期检测、自动清理）

**三个核心组件的职责分工：**

```
SessionManager（会话管理器）
  └─ 负责：会话生命周期 → 创建/获取/关闭/删除/过期清理
  └─ 设计：单例模式 + 线程安全（threading.RLock）+ 后台清理线程

DialogueTracker（对话跟踪器）
  └─ 负责：意图识别 → 知识检索/记忆查询/一般问答/报告生成/需要澄清
  └─ 负责：实体提取 → 城市、月份、用户ID、数字等
  └─ 负责：槽位填充 → 收集对话中逐步获取的信息

IntentParser（指令解析器）
  └─ 负责：结构化指令解析 → 搜索/生成/总结/比较/解释
  └─ 负责：修饰词提取 → 时间范围、数量限制、格式要求
```

**会话过期的设计：**

会话默认 1 小时超时（`session_timeout=3600`）。每次 `get_session()` 时检查 `updated_at + timeout < now`，过期则抛 `SessionExpired` 异常。后台线程每 5 分钟扫描一次过期会话并自动清理，防止内存泄漏。

**意图识别的实现方式：**

基于关键词 + 正则匹配的轻量方案（而非用 LLM 识别）：

```python
if "报告" in query:        return "report"          # 报告生成
if "记得" in query:        return "memory_query"     # 记忆查询
if "解释" in query:        return "knowledge"       # 知识检索
if "你好" in query:        return "general"          # 一般问答
```

**为什么不用 LLM 做意图识别？**

意图识别需要在 Agent 推理之前完成（用来决定是否切换提示词），而 Agent 本身也是用 LLM 推理的。如果意图识别也调 LLM，就是"先用一次 LLM 判断意图，再用 Agent 推理执行"，增加了一倍延迟。关键词匹配虽然不如 LLM 灵活，但延迟接近零，在这个场景下足够了。

---

### 3.2 名词解释

**Session（会话对象）**
一个状态机，包含：
- `session_id`：唯一标识
- `messages`：消息列表 `[{role: "user", content: "..."}, ...]`
- `state`：`DialogueState` 对象（意图、实体、槽位、轮次计数）
- `created_at` / `updated_at`：用于过期检测
- `status`：active / expired / closed

**DialogueState（对话状态）**
- `intent`：当前识别到的意图
- `entities`：已提取的实体（如 `{"city": "深圳", "month": "6月"}`）
- `slot_values`：已填充的槽位（如 `{"report_month": "6月", "report_type": "使用报告"}`）
- `turn_count`：当前对话轮次
- `last_tool`：上一次调用的工具名
- `pending_clarification`：是否需要向用户追问澄清
- `requires_follow_up`：是否需要补充提问

**槽位填充（Slot Filling）**
对话系统中收集信息的机制。比如生成报告需要知道"哪个月"和"什么类型的报告"，这两个就是槽位。用户说"生成 6 月使用报告"，DialogueTracker 提取出 month=6月、type=使用报告，填充到槽位中。如果用户只说"生成报告"没说月份，`pending_clarification=True`，Agent 追问"请问需要哪个月的数据？"

**上下文注入**
调用 Agent 之前，从 SessionManager 获取最近 N 条消息（`get_recent_context(n=10)`），格式化为 LangChain 兼容的 `{"role": "user/assistant", "content": "..."}` 列表，注入到 Agent 的 `input_dict["messages"]` 中。Agent 在推理时就能看到完整的对话历史。

---

### 3.3 面试追问及回答要点

**Q：为什么用内存存储会话，而不是 Redis/数据库？**

> 这是一个分层演进的问题。当前是个人项目，内存存储的优点：① 零依赖——不需要额外部署 Redis 或数据库；② 读写极快——纯内存操作，纳秒级延迟；③ 代码简单——一个 dict + 线程锁就够了。
>
> 缺点也清楚：服务重启会话丢失（单机部署可以接受），多副本部署不能共享会话（当前没有多副本需求）。
>
> 如果后续需要演进——用 Redis 替代内存 dict，Session 对象序列化为 JSON 存入 Redis，设置 TTL 自动过期。代码改动很小（SessionManager 的接口不用变，只改存储实现）。

**Q：上下文窗口有限（比如 qwen3.7-max 是 128K），对话太长怎么办？**

> 当前设置了 `max_context_messages=10`（最多只传最近 10 条消息给 Agent），所以对话随怎么聊都不会撑爆窗口。但这意味着第 11 条之前的消息就"被遗忘"了。
>
> 这时候长期记忆就发挥作用了——关键信息被 MemoryExtractor 提取并存到长期记忆后，即便消息历史截断，Agent 通过 `search_memory` 工具还能找回之前的上下文。短期记忆负责"最近在聊什么"，长期记忆负责"之前聊过的关键信息"。
>
> 进一步优化可以用"摘要压缩"——把第 11+ 条消息用 LLM 压缩成一段摘要（而非直接丢弃），但当前项目没有实现。

**Q：DialogueTracker 的关键词匹配会不会误判意图？**

> 会的。比如用户说"上次报告说错了"，包含"报告"关键词，但意图不是"生成报告"而是"纠正"。关键词匹配确实有误判率。
>
> 当前通过两个手段缓解：① 多个关键词组合判断（而非单一关键词）；② 意图只是给 Agent 一个"建议"，Agent 本身还有 LLM 的语义理解能力做最终决策。Agent 收到意图标记后，如果觉得不对，可以自行调整。
>
> 面试中被问到这个问题，关键是展示你清楚这种方案的局限性和改进方向，而不是硬说它完美。

## 技术点 4：RAG 检索评估体系（6 项指标 + 自动测试集）

> 简历原文：*建立 6 项指标的 RAG 检索评估体系（Recall / Precision / MRR / NDCG / F1 / Hit Rate），支持从知识库自动生成测试数据集，量化衡量检索策略改进效果，解决了 RAG 调优只能凭感觉、无法量化的问题。*

---

### 4.1 设计思路

**问题**：RAG 系统有很多可调参数——chunk_size、chunk_overlap、embedding 模型、top_k 值、检索策略（纯向量 vs 混合检索）。改了参数后，怎么知道效果变好了还是变坏了？靠肉眼判断几个案例不可靠。

**方案**：建立一套自动化的评估体系，用可量化的指标衡量检索质量。

**整体流程：**

```
1. 自动生成测试集 → 从知识库所有文档中提取主题，用模板生成问题
       ↓
2. 运行评估     → 对每个测试问题，调用检索服务，获取 Top-K 结果
       ↓
3. 计算指标     → 比较检索结果与 ground_truth，计算 6 项指标
       ↓
4. 生成报告     → 指标汇总 + 失败案例详情
```

**测试集的自动生成策略（BenchmarkDataset）：**

不需要人工标注。从知识库中分析文档，提取主题关键词，然后模板化生成测试问题：

```
从文档名和 Markdown 标题中提取主题 → ["GIL", "装饰器", "面向对象", ...]

使用 10 种问题模板生成问题：
  - "{topic}是什么"        →  "GIL是什么"
  - "请解释{topic}"         →  "请解释装饰器"
  - "{topic}的作用"         →  "面向对象的作用"
  - "{topic}的定义"         →  "GIL的定义"
  - ...

ground_truth: 通过向量搜索找到与问题最相关的文档块 ID 作为正确答案
```

**评估执行（RAGEvaluator）：**

```python
for query, ground_truth_doc_ids in test_cases:
    retrieved_docs = vector_store.search(query, top_k=k)
    retrieved_ids = [doc.metadata["source"] for doc in retrieved_docs]

    # 对每个 Top-K 值计算指标
    recall_at_3 = recall_at_k(retrieved_ids[:3], ground_truth_doc_ids)
    precision_at_5 = precision_at_k(retrieved_ids[:5], ground_truth_doc_ids)
    mrr_value = mrr(retrieved_ids, ground_truth_doc_ids)
    # ... 最后汇总所有测试用例取平均
```

---

### 4.2 名词解释

**Recall@K（召回率）**
```
Recall@K = 检索结果中相关文档数 / 知识库中总共的相关文档数
```
回答"找全了吗？"的问题。Recall=1.0 表示所有相关文档都被检索出来了。比如知识库中有 5 个文档和"GIL"相关，检索 Top-3 命中了 2 个，Recall@3 = 2/5 = 0.4。

**Precision@K（精确率）**
```
Precision@K = 检索结果中相关文档数 / K
```
回答"找对了吗？"的问题。Precision=1.0 表示检索出来的都是相关的，没有"噪音"。比如检索 Top-5 中有 3 个相关，Precision@5 = 3/5 = 0.6。

**注意**：Recall 和 Precision 往往此消彼长——K 越大 Recall 越高（找回了更多），但 Precision 可能下降（混入了更多不相关的）。

**MRR（Mean Reciprocal Rank，平均倒数排名）**
```
MRR = 1 / 第一个相关文档的排名
```
回答"第一个对的排在什么位置？"的问题。第一个相关结果排在第 1 位，MRR = 1.0（最好）；排在第 10 位，MRR = 0.1。MRR 越高说明系统能把最相关的结果排在越前面。这个指标对搜索场景很重要——用户通常只看前几个结果。

**NDCG@K（Normalized Discounted Cumulative Gain）**
```
DCG = Σ (2^relevance_score - 1) / log2(rank + 1)
NDCG = DCG / IDCG（理想排序下的 DCG）
```
更精细的排序质量指标。它不仅考虑"对不对"，还考虑"排序好不好"——高相关性的文档排前面得分高，排后面得分低。相比于 MRR 只关心第一个对的排在哪，NDCG 关心的是整个排序列表的质量。

**F1 Score**
```
F1 = 2 × Precision × Recall / (Precision + Recall)
```
精确率和召回率的调和平均。两者都重要但方向相反时，用 F1 来权衡。F1 高说明检索既没有漏掉太多（Recall 还行），也没有混入太多噪音（Precision 还行）。

**Hit Rate（命中率）**
```
Hit Rate = 至少命中一个相关文档的查询数 / 总查询数
```
回答"至少找到一点吗？"的问题。一个宽松但实用的指标——如果 Hit Rate 只有 0.3，说明 70% 的查询连一个相关文档都找不到，检索策略有严重问题。

**Ground Truth（标准答案/基准真相）**
评估时的"正确答案"。在 RAG 评估中，ground truth 是"哪些文档和这个问题相关"。当前项目通过自动生成的方式构建 ground truth（用向量搜索找到最相似的文档作为相关文档）。

**BenchmarkDataset（基准测试数据集）**
从知识库自动生成的测试用例集合。不需要人工标注，降低了评估的门槛。局限性是：自动生成的 ground truth 不一定 100% 准确（可能会有遗漏或标错），但对于比较不同参数的相对效果已经够用了。

---

### 4.3 面试追问及回答要点

**Q：为什么选这 6 个指标？是不是越多越好？**

> 6 个指标覆盖了检索质量的不同维度，每个回答不同的问题：
>
> - Recall — "找全了吗"（适合回答"不要漏东西"的场景）
> - Precision — "找对了吗"（适合回答"不要有噪音"的场景）
> - MRR — "第一个对的排在哪"（适合回答"搜索体验好不好"的场景）
> - NDCG — "整体排得怎么样"（比 MRR 更精细）
> - F1 — "综合平衡好不好"（当你需要在 Recall 和 Precision 之间权衡时）
> - Hit Rate — "至少能找到吗"（最基础的底线指标）
>
> 不代表越多越好——要选对场景的指标。比如对于 FAQ 场景（用户只关心第一个答案对不对），MRR 和 Hit Rate 最重要；对于知识检索场景（需要多个文档综合回答），Recall@5 和 NDCG@5 更重要。

**Q：自动生成的测试集靠谱吗？会不会自己测自己？**

> 诚实地说——确实有一定 bias。因为 ground truth 是用向量搜索找的，然后用同样的向量搜索去评估，会有"自己证明自己是对的"的风险。
>
> 但它的价值在于横向对比——我在调 chunk_size 之前和之后各跑一次评估，看指标是涨了还是跌了。即便绝对值不一定精确，**相对变化方向是可信的**。
>
> 如果要更严格：找几个真实问题，人工标注 ground truth，作为验证集；或者用 LLM 辅助标注（让 LLM 判断检索结果是否相关），比纯向量搜索的 ground truth 更可信。

**Q：调参时，你是怎么看这几个指标的变化来决策的？**

> 举一个实际例子：我把 chunk_size 从 200 改到 500，跑了一次评估，发现 Recall@3 提高了（chunk 大了，更容易命中），但 Precision@3 下降了（chunk 大了，不相关的内容也混进去了），MRR 基本不变。
>
> 决策：如果这个项目的使用场景是"用户需要精确的一个知识点答案"，Precision 更重要，那就维持 chunk_size=200。如果场景是"用户想看相关的多个段落"，Recall 更重要，那就用 chunk_size=500。
>
> 这回答了面试官想知道的——你不只是调参，你能读懂指标并做业务决策。

**Q：如果让你改进这个评估体系，你会怎么做？**

> ① 人工标注一个小规模验证集（50-100 条），用 LLM 辅助判断检索结果的相关性，解决自动生成 ground truth 的偏差问题；② 增加"答案质量评估"指标（不只是检索好不好，还有最终生成的回答好不好），可以用 LLM-as-Judge（让另一个 LLM 打分）、RAGAS 框架；③ 把评估集成到 CI/CD 中——每次 push 代码或改配置，自动跑评估，指标下降就报警。
>
> 这些都是加分项，展示你有工程化思维。

## 附录：面试通用技巧

### 回答 STAR 法则

面试回答技术问题，建议用 STAR 结构：

- **S**ituation："项目中有个问题——ChatGPT 不知道我们自己的知识库内容"
- **T**ask："我需要实现一个系统，让大模型能基于我们的文档回答"
- **A**ction："我设计了 RAG 架构，用 Chroma + text-embedding-v4 做了向量检索链路"
- **R**esult："解决了大模型胡编的问题，回答有据可查；评估体系量化验证了效果"

### 展示"知其然也知其所以然"

不要只背答案。面试官喜欢听你说"为什么做这个选择"而非"我做了什么"：

- ❌ "我用了 SQLite + Chroma 双写"
- ✅ "Chroma 擅长语义检索但分类查询弱，SQLite 补充了这个短板，两者互补实现了完整的记忆搜索能力"

### 准备"踩过的坑"

每个技术点至少准备一个你实际遇到的问题和解决过程，这比背书更打动面试官：

- RAG：chunk_size 调小了 Recall 下降，调大了 Precision 下降，找到平衡点
- 记忆：LLM 有时候把闲聊也当成重要记忆提取，后来加了 importance 过滤
- 评估：自动生成的 ground truth 有偏差，但用于横向对比参数效果足够了
